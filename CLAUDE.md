# CISC 699 — Applied Project in Computer Information Sciences

This directory holds the student's capstone project for **CISC 699 (SU26, 3 credits)** at Harrisburg University, M.S. in Computer Information Sciences. Instructor: Dr. Majid Shaalan. Term runs 2026-05-09 through 2026-08-14.

CISC 699 is the culminating applied-project studio of the M.S. program. The deliverable is **one substantial computational solution** to a meaningful technical, professional, organizational, or societal problem, plus a graduate-level technical report. It is **not** a tutorial replication, product-assembly exercise, or unscoped build. Individual project by default.

## Required deliverables (graded weights)

| Deliverable | Weight |
|---|---|
| Approved proposal and scope package | 10% |
| Design review / literature / evaluation plan | 15% |
| Midpoint progress review or demo | 10% |
| Final computational artifact and technical package | 30% |
| Final technical report | 25% |
| Final presentation and defense/demo | 10% |

## 14-week milestone map

- **W1** — Orientation; problem framing; charter draft; initial bibliography.
- **W2** — Proposal approval; success criteria finalized; project plan submitted.
- **W3** — Literature/related-work synthesis; requirements, use cases, data sources.
- **W4** — Architecture/design review; methods, environment, evaluation plan approved.
- **W5–W6** — Implementation sprints I–II; repo setup, documentation baseline, data/toolchain validation, early evidence.
- **W7** — Midpoint demo / progress review; scope adjustment if needed.
- **W8** — Implementation sprint III; deeper testing, benchmarking, comparative analysis.
- **W9** — Results, limitations, error analysis, broader-impact draft.
- **W10** — Artifact hardening; README, setup, reproducibility materials.
- **W11** — Full draft of final technical report due for feedback.
- **W12** — Revision cycle; deck and demo script.
- **W13** — Presentation rehearsal; final package check.
- **W14** — Final report, artifact, and presentation submission; reflection.

## Standards for the final submission

- **Functioning computational artifact** appropriate to the approved scope, with a repository link or submission package.
- **Final technical report**, typically **15–25 pages excluding appendices**, graduate-level English, consistent citations, figures, tables.
- **README** with setup, dependencies, execution steps, I/O expectations, and known limitations.
- **Versioned source** — source files, scripts, notebooks, configs, test assets, anything needed to understand or reproduce the work.
- **Testing / validation evidence** — unit/integration tests, benchmarks, ablation or comparison results, screenshots, logs, user feedback, analytical notes.
- **Data statement** — provenance, licensing, privacy constraints, synthetic/public/proprietary/anonymized decisions.
- **AI-use disclosure appendix** whenever generative AI was used for ideation, coding assistance, debugging, drafting, or analysis.

## Final report structure

1. Title and abstract
2. Problem statement and significance
3. Related work / background
4. System or solution design
5. Implementation details
6. Evaluation methodology
7. Results and analysis
8. Ethics, security, privacy, accessibility, and broader impact
9. Limitations and future work
10. References and appendices

## Evaluation dimensions the work is graded on

- **Problem framing & significance** — worthwhile, realistically scoped, grounded in the field.
- **Technical design rigor** — defensible choices on architecture, algorithms, data, tools, interfaces, testing.
- **Implementation quality** — appropriate depth, engineering competence.
- **Evaluation & evidence** — claims backed by experiments, tests, or analytical evidence.
- **Communication quality** — organized, clear, professional, technically accurate.
- **Ethics & broader impact** — privacy, security, accessibility, societal implications addressed.
- **Professional process** — milestones managed, feedback acted on, work documented reproducibly.

## Working conventions for this repo

- **Git-based version control** is expected; maintain a clean history and meaningful commits as evidence of process.
- Keep an **issues / risk log** alongside the code.
- Track **reproducibility**: pinned dependencies, deterministic setup, clear run instructions.
- Maintain testing/evaluation artifacts in-repo (test suites, benchmark scripts, result tables, logs).
- Treat the **README** as a living document — it is a graded submission requirement, not an afterthought.

## AI-use policy (binding constraint on this project)

- AI-generated text, code, analysis, or figures may **not** be submitted as wholly unaided work.
- **All substantial AI assistance must be disclosed** in the AI-use appendix of the final report.
- Do **not** feed confidential, proprietary, personal, FERPA-protected, HIPAA-regulated, or otherwise restricted information into public AI systems without formal authorization.
- The student remains fully responsible for correctness, originality, legality, and appropriateness of every submitted artifact.

When Claude assists in this repo, keep a running note of what was generated vs. authored, so the disclosure appendix can be assembled accurately at the end.
