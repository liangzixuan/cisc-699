## What this is

LLM-powered software agents increasingly execute model-generated code as part of their tool-use loop. Most deployments rely on isolation designed for trusted multi-tenant workloads rather than for adversarial or malfunctioning code generated by a language model. This project builds a hardened Python execution sandbox suitable for LLM agent tool-use, with progressively stronger isolation back-ends (hardened Docker, then gVisor) and a curated **adversarial evaluation suite** that allows the isolation strength of the sandbox to be measured empirically rather than asserted.

See [`docs/01-launch-packet/problem-statement.md`](docs/01-launch-packet/problem-statement.md) for the full framing.

Manual CLI example:

```bash
PYTHONPATH=src python3 -m safeexec --backend local --code "print('hello')"
```

```bash
curl -s http://127.0.0.1:8080/execute \
  -H 'Content-Type: application/json' \
  -d '{"code":"print(2 + 2)"}'
```

## Evaluation (planned)

The artifact's claims will be supported by three test suites and a documented threat model:

1. **Functional suite** — ≥100 programs whose output must match a reference Python interpreter.
2. **Adversarial suite** — ≥40 curated programs across categories (resource exhaustion, fd exhaustion, fork bombs, syscall abuse, attempted persistence, host enumeration, exfiltration attempts). Each carries an expected contained-outcome label.
3. **Performance benchmarks** — cold-start, warm-start, and steady-state latency; memory overhead; throughput. Reported with confidence intervals.

Each suite is run under each isolation back-end (hardened Docker, gVisor). Full methodology will live in `docs/design/evaluation-plan.md`.
